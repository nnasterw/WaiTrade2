# V9.1 — 1m Bounce Confirmation Entry

发布日期: 2026-05-08

## 关键变动 (vs V8.5)

- 入场方式从"5m触及OB即挂限价单"改为"1m反弹30%确认后市价入场"
- 信号保留率: 97.7% (过滤2.3%穿越型，这些在live必亏)
- 预期live胜率: 98.2% (vs V8.5回测 97.6%, vs V8.5 live实际 50-67%)
- 复利效率: V8.5的96.7%

## 核心逻辑

```
5m bar 生成V8.4信号 → 信号进入"待确认"状态
  → 每60s轮询1m bars
    → 任意1m bar收盘反弹 ≥ 30%×OB高度 → 确认，市价入场
    → 10分钟无反弹 → 过期丢弃
```

## 为什么这样做

回测中V8.5看完整5m OHLC判断成交，但live只有实时tick。价格在5m内短暂穿越OB(无反弹)时：
- 旧方案(限价单): 成交了，然后直奔止损 → 这是12败1胜的原因
- V9.1(等反弹): 不成交，避免了穿越型亏损

## 确认标准

```python
# long OB: 1m bar close - entry_price >= 30% × (ob_top - ob_bottom)
# short OB: entry_price - 1m bar close >= 30% × (ob_top - ob_bottom)
# 窗口: 10分钟 (约10根1m bar)
# 确认后: 检查市价偏移，< 0.5R才入场
```

## 入场价偏移保护

确认后检查当前市价与理想入场价偏移：
- 偏移 < 0.3×risk → 正常入场
- 偏移 0.3-0.5×risk → 入场(但盈亏比略差)
- 偏移 > 0.5×risk → 放弃(价格走太远)

## 主循环行为

- 无待确认信号时: 正常5分钟轮询
- 有待确认信号时: 自动切换60秒短轮询
- 10分钟超时后: monitor自动清理，恢复正常轮询

## 与V8.4/V8.5的关系

- 信号生成: 完全复用V8.4的generate_v84_live_limit_candidates
- OB识别/过滤/分层/止损: 完全不变
- 唯一区别: 入场执行方式(限价单 → bounce确认+市价)

## 实现细节

- `BounceMonitor` dataclass 跟踪每个信号的确认状态
- `check_bounce_confirmations()` 每轮检查1m bars
- 多级入场(mid/deep/bottom)共享确认: OB有效=所有层级同时入场
- 重启不需要恢复state: monitor最多存活10分钟
