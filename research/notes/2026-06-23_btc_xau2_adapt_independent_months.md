# 2026-06-23 BTC XAU2 思路适配与独立月复核

## 目标

将 XAU2 的“尾部截断、顺势持仓、反手修复”思路适配到 BTCUSDm，并用 MT5 Strategy Tester CLI / Model 4 / Real Ticks 验证 24 个独立月。

本轮复核严格区分两个口径：

- 独立月：每个月单独从 $200 启动。
- 连续 720 天：同一账户路径持续复利。

连续复利口径不能证明独立月达标。

## 固化策略

新增策略键：

`v12btc-xau2-pos24`

继承自：

`btc_v11_tailcap_dtp2_swp_combo_l01_lock1_h4pos1_rev_no_dtp`

核心机制：

- BarTF=M5，降低 M1 噪音。
- max_lot_size=0.01，max_pos_mult=1.0，保持小资金稳定。
- H1 net push 顺势加权，逆势降权。
- no-MFE 失败后允许反手修复。
- 低余额 OB 坏月份/坏小时过滤。
- 月度锁盈保留，避免小盈利月回吐。

`python scripts\check_strategy_consistency.py v12btc-xau2-pos24`

## 现有候选池上界

扫描 `results/backtest/*BTCUSDm*24m_html*.csv` 后发现：

- 完整 24 独立月里，高收益腿 `v11_r61_j2_g30_shallow_cut_counter_risk14` 只有 3/24 个月 >= $300，15/24 盈利。
- 稳定腿 `v12btc-xau2-pos24` 为 24/24 盈利，但无月份 >= $300。
- 即使按每个月后验选择已有候选池最优结果，也只有 5/24 个月 >= $300。

因此，现有参数组合无法实现“24 个独立月每月 +$300”。继续靠风险放大或月度后验过滤会破坏独立月稳定性。

## 下一步方向

必须新增可实时识别的结构性利润来源，而不是继续放大现有小收益腿：

1. 大周期 OB / 供需区触达后的趋势腿，优先 H1/H4，类似 XAU 2605 的大周期 OB 压制逻辑。
2. 入场周期从 M1 提升到 M5/M15，使用小周期结构突破和 K 线动能延续确认，减少噪音扫损。
3. 出场遵循“未出现小周期反转、动能未衰弱到强力反弹、反向动能未连续”则继续持仓。
4. 只在实时可观察的 HTF 顺势、结构突破、强弱高低点转换、流动性 sweep 后确认延续时放大仓位。
5. 保留 `v12btc-xau2-pos24` 作为稳定底仓，对新增趋势腿做独立风险隔离。

2. 失败反手只在小周期反向结构突破 + 反向净动能延续时触发，普通 no-MFE 只记录失败状态。
3. 针对 2024-10、2026-04、2026-05 的大亏单做订单级特征抽取，按实时特征拆分：HTFPB 本体、失败反手、普通 OB 是否分别亏损。

## 2026-06-23 追加：目标纠偏与高容量月目标实验

目标重新校准：用户要求不是“24 独立月转正”，而是“24 个独立月每月盈利至少 $300”。因此 `v12btc-xau2-h1pb20-fre-mom5-probe2-add-pt15` 虽然达到 23/24 盈利，但它使用 1.5% 月盈利停止，约 +$3 即停止，天然不可能满足每月 +$300。

本轮重点验证：把月盈利目标改为 150%（$200 账户即 +$300），并提高容量，观察是否可以让月内先触达 +$300 后锁定。

重要实现发现：

- `UseBTCProfile()` 只有在 `InpEnableBTCProfile=true` 且品种名包含 `InpBTCProfileSymbol` 时才启用。
- 当前最优 `pt15` 的 set 未写 `InpEnableBTCProfile=true`，实际走通用参数 `InpRiskPercent/InpMaxLotSize/InpMaxPosMult`，不是 `InpBTC*`。
- 因此本轮分两类验证：通用 profile 高容量，以及显式 BTC profile 高容量。

订单级/结构级结论：

- 当前信号族可以产生单月 +$300，但无法让所有独立月稳定触达 +$300。
- 统一放大普通底仓会把 2024-10 这类方向错链放大；HTFPB-only 放大能吃到 2025-12/2026-03，但在弱结构月直接打到月亏损止损。
- 这证明问题不在“月目标停止”或“仓位容量”，而在更高置信度结构信号缺失：必须先识别 H1/H4 级别强高低点、结构突破/回踩、供需区/OB 反应，再决定是否允许高容量。
- `pt15` 可保留为“小盈利稳定版”参考，但它不是达成每月 +$300 的路线。

下一轮应新增代码级结构信号，而不是继续参数放大：

1. H1/H4 结构级别持久化：记录强高低点、BOS 方向、突破价、回踩区间、触发次数。

## 2026-06-23 追加：HTF BOS 回踩结构腿验证

本轮新增 WaiTrade2 代码级 `HTF BOS 回踩` 补充区，目的是把 XAU2 的“趋势中小周期不反转且动能未衰弱就持有”思想，迁移成 BTC 可实时识别的 H1/H4 结构信号。新增信号不是按月份筛选，而是基于实时特征：H1/H4 swing 高低点、收盘突破缓冲、突破后最小延伸、回踩区半宽、M5 入场动能确认。

实现要点：
- 新增 `InpHTFBOSRetest*` 参数族，避免与 WaiTrade3 历史 `InpBOSRetest*` 重名。
- `OBZone` 增加 `is_bos_retest`，BOS 区不参与普通 OB 合并，独立 TTL。

## 2026-06-23 追加：HTF OB 共振与探针加仓验证

本轮继续围绕“高容量只能绑定实时结构确认”推进，验证两个可证伪假设：

1. H4 BOS 回踩如果要求附近存在同向 HTF 订单块/供需区，是否能过滤掉短持仓大亏。
2. H4 BOS 首单只做 0.01 探针，只有跑出顺向 MFE/强动能后再按风险重新加仓，是否能保留趋势利润并切掉早期亏损。

代码改动：
- 新增 `InpHTFBOSRequireOBConfluence`、`InpHTFBOSOBLookbackBars`、`InpHTFBOSOBToleranceATR`、`InpHTFBOSOBMinImpulseATR`。
- BOS 创建前在同一 HTF K 线数组内查找“反向 K + 同向冲动”的 HTF 订单块，要求 BOS level 与订单块距离小于 ATR 容差。
- 新增 `InpStrongAddOnUseRiskLot`、`InpStrongAddOnRiskPercent`、`InpStrongAddOnMaxLotSize`、`InpStrongAddOnFamilies`，允许探针单确认后按风险百分比重新计算加仓手数，而不是只按 0.01 探针手数倍增。

- H4 BOS-only、H4 BOS+OB 共振、H4 BOS 探针加仓均未接近“24 独立月每月 +$300”。
