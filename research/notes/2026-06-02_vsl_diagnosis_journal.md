# VSL 诊断日志 — 2026-06-02 全流程

## 核心问题

0529 回测：QS3 亏损 $12.89，34 笔交易。用户观察到 OB 有效但入场被影线刺破、BE 锁过早踢出。

## 五版 VSL 迭代

### v1: 券商 SL + hard_buffer (0.15R)
**假设**: 把实 SL 放宽 15%，给价格呼吸空间。
**结果**: 720d $282K → $118K (-58%)
**根因**: hard_buffer 让损失扩大 15% 但盈利不变 → R/R 从 1.50 恶化到 1.30。`BrokerStopFromVirtualSL` 用 virtual_risk 算 hard_buffer，但 lot 仍用 virtual_risk → 实际 $风险 = 配置的 1.15 倍。

### v2: lot 补偿 + TP 补偿
**假设**: lot 基于 broker 风险重算，TP 补偿 hard_buffer。
**结果**: 720d $98K (-65%)
**根因**: $200 账户 lot 已是最小 0.01，仓位调整不可见。TP 补偿（更远目标）反而降低了命中率。

### v3: 软止损 (K线收盘确认, confirm_bars=1)
**假设**: 不设券商 SL，EA 等 K 线收盘确认后才平仓。影线（收盘前收回）→ 不触发；真突破（收盘确认）→ 平仓。
**结果**: 0529 +$2.40，720d $1.8K (-99%)
**根因**: M1 K 线 = 60 秒。真突破等 60 秒再平仓 → 填价严重恶化。60 秒在 XAU 上可以移动 0.3-0.5pt → 每笔多亏 0.15-0.25R。720 天真突破远多于影线 → 累积亏损灭顶。

### v4: 秒级确认 (breach_sec=5) — **突破**
**假设**: 不用 K 线收盘，用秒级时间确认。影线 <1s，真突破 >5s。5 秒延迟比 60 秒快 12 倍。
**实现**: `CheckVirtualSLBreach()` 每 tick 检查。价格突破 virtual_sl → 计时 → 持续 5s → 平仓。价格收回 → 清零。
**结果**: 0529 +$6.07 (WR +10%, PF 2x)，720d $224K (v1 的 79%)

### v5: WIDE_SL (sl_buffer_atr=0.5)
**假设**: SL 放宽 0.2 ATR ≈ 0.5pt，覆盖 0529 所有 13 笔 <0.26pt 的穿透。
**结果**: 0529 +$2.52，不如 VSL_5s (+$6.07)。WIDE+VSL 组合反而不如单独 VSL。

## 关键发现：VSL + BE 锁互杀

**0529 13 笔全亏全部 <0.26pt 穿透 → VSL 应救活大部分 → 但只见到 +$6.07。**

追踪一笔被 VSL 救活的交易：

```
入场 → 影线下刺 0.1pt → VSL 等 5 秒 → 影线收回 → 存活!
  → 涨到 0.5R → BE 触发 → SL 移到 0.4R 锁
  → 微回落 0.1R → 触及 BE 锁 → +0.4R 平仓 ← 又被杀了!
  → 价格后来涨到 1.5R TP...但仓位已不在
```

VSL 救活的交易只赚 +0.4R（BE 锁），不是 +1.5R（TP）。**每笔被救的交易被 BE 锁吃掉了 1.1R 的潜在利润。**

数学模型：
- 4 笔影线被 VSL 救活 → BE 锁在 0.4R → +0.4R each = +1.6R
- 9 笔真突破 VSL 延迟 5s → -1.1R each = -9.9R
- 净 = -8.3R → vs 无 VSL (-13R) = +4.7R ≈ +$9.4（接近实测 +$6.07）

如果被救交易不受 BE 锁限制：
- 4 笔到 TP +1.5R each = +6.0R（vs BE 锁的 +1.6R → 被 BE 吃掉了 +4.4R！）

## 修复方案

**VSL 救活的交易 → 标记 `survived_breach` → BE 推迟到 1.0R/0.6R**

让被 VSL 证明 OB 有效的交易有更大呼吸空间，不被 BE 锁过早踢出。

改动：
1. `Types.mqh`: PosTrack 加 `survived_breach` 标记
2. `PositionManager.mqh`: CheckVirtualSLBreach 发现 price 收回后设标记
3. `PositionManager.mqh`: CheckBreakeven 检查标记 → 用延迟 BE 参数

## 教训

1. **反馈回路是一切**：A/B 回测对比（20 秒/次）让快速试错成为可能
2. **不要相信单次测试**：workflow 报告的 +$6.07 需要在不同环境下复现
3. **修改参数 ≠ 修改代码**：WIDE_SL 是参数改动，VSL 是代码改动，两者效果不可直接比较
4. **组件间交互是隐藏的 bug**：VSL 和 BE 锁各自正确，但组合产生意外行为
5. **数学推导先于代码**：VSL+BE 互杀的数学推导解释了为什么改善不如预期
